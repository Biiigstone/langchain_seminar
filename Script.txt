자기소개
안녕하세요 이번에 랭체인을 주제로 세미나를 진행할 장신원이라고 합니다. 
꽤 오랜 시간 동안 뜨거운 감자인 LLM의 작동 원리를 간단히, 그리고 이를 활용할 수 있도록 돕는 랭체인 프레임 워크에 대해서 알아보겠습니다.
그래서 이제~ 배경과 왜 랭체인을 사용하는지, 랭체인의 주 구성요소들은 무엇이 있는지, 그리고 LCEL을 어떻게 선언하고 동작하는지에 대해 알아보겠습니다.

Why?(왜 랭체인이 등장하게 되었는가)

배경 설명(BackGround)


1) LLM 작동 방식&원리(simple ver)
  텍스트의 이해: Attention(QKV)를 활용한 트랜스포머 모델 기반. 
 LLM의 원리 하면 가장 먼저 생각나는 것이 있죠~ 다들 알고 계시겠지만, 트랜스포머와 어텐션 일 것입니다.
 LLM은 텍스트를 어텐션을 사용하여 이해합니다. 
  예를 들어
  예시 : 신원은 개를 키운다. 그의 개가 다른 사람의 개를 물었다.
  이러한 문장이 있을 때, 사람은 보자마자 뒷문장의 그가 어떤 사람을 지칭하는지, 각 개가 어떤 개를 지칭하는지 이해합니다. 어텐션이 없는 Word2Vec 등은 대명사 그가 가리키는 존재가 누구인지 알지 못한다. 마찬가지로 문장 내의 '개'는 모두 같은 '개'라는 의미(수학적 벡터)일 뿐, 동사의 목적어가 되는 '개'와 주어가 되는 '개' 도 구분하지 못한다.
  Attention은 이러한 맥락 문제를 Query Key Value 행렬과 내적을 통해 해결한다.
  즉, Attention은 LLM에게 텍스트 이해력을 부여한다.
  
  텍스트의 생성: AutoRegressive
  자신의 과거 출력이 미래 출력을 결정하는 방식. 예를 들어 첫 단어가 "나는"이라면 뒤에는 "~을"과 같은 목적어가 높은 확률값을 가질 것이며, 이를 기반으로 "너를" 이라는 결과를 도출합니다. 자기 회귀는 이 출력을 다시 입력으로 넣습니다. "나는 너를" 다시 입력하며, 이어서 "~하다" 라는 동사가 높은 확률값을 가질 것이다. 최종적으로 "나는 너를 좋아한다" 라는 문장이 완성됨. 자기 회귀 모델의 역할은 논리적 추론이 아니라 "확률적 연결"에 의한 텍스트 생성기라고 할 수 있음.
  즉, AutoRegressive는 LLM에게 텍스트 생성력을 부여한다.
  
2) LLM은 결국 "그럴듯한 텍스트 생성기"

그렇다면 LLM은 결국 무엇일까요? 

 구름 낀 여름 하늘날을 보며, LLM에게 맥락 배경 없이, LLM에게 '오늘은 구름이 많네, 오늘 날씨가 어떻게 될까?' 라고 질문한다면 LLM의 사전 학습된 지식(pre-trained knowledge)에 기반하여 '구름이 많다 => 비가 올 확률이 높다'로 연결되어 실제 날씨 정보와 관련 없이 그저 그럴듯한 답을 내놓을 뿐임. 

  결론적으로 LLM은 고도의 논리 엔진이 아니다. 앞서 서술하였듯, 과거의 단어를 바탕으로 다음 단어가 올 "확률"을 계산하여 이어 붙이는 그럴듯한 텍스트 생성기이다.

 즉, LLM은 다음과 같은 한계를 지닌다
 - 정보의 단절(Cut-off 존재) : 모델의 가중치는 자신이 학습한 데이터에 한정된다. 최신의 정보는 반영하지 못한다.
 - 행동 능력의 부재 : LLM은 텍스트만 생성할 뿐 실제로 DB에 쿼리를 날리거나 웹에 검색할 수 있는 능력이 없다.
 - Stateless : LLM은 매번 단어 확률을 게산할 뿐, 이전 대화의 맥락을 자체적으로 기억하지 못한다.
 최종적으로 이러한 한계들은 Hallucination 현상의 발현으로 이어진다.
 
3) Why LangChain?

 이와 같이 LLM은 그 자체로 뛰어난 자연어 이해 및 생성 능력을 갖추고 있으나, 앞서 언급한 구조적 결함으로 인해 Standalone 시스템으론 한계를 가진다. 이를 극복하고 활용하기 위해서 RAG(Retrieval Augmented Generation), ReAct(Reasoning & Acting) 와 같은 여러 패러다임이 제시되었으며 이를 일관된 인터페이스로 구현할 수 있도록 돕는 것이 LangChain 프레임워크이다.
 
 LangChain 프레임워크는 문맥 기억, 최신 혹은 외부 정보의 주입을 용이하게 할 뿐만 아니라 Chain 기반의 연결을 통해 단계적 사고를 가능케 한다.
 LangChain 프레임워크 위에서 LLM은 더 이상 확률적 텍스트 생성기가 아니라, 외부 환경과 상호작용하며 복잡한 태스크를 수행하는 논리적인 추론 엔진으로 작동할 수 있다.

What? 랭체인의 주요 구성요소

앞서 LLM의 본질적 한계와, 이를 제어하기 위한 프레임워크로서 LangChain의 도입 배경을 확인했습니다.
개별 컴포넌트의 기술적 명세를 보기에 앞서, 데이터가 시스템 내에서 어떻게 흐르는지 전체 흐름도를 보겠습니다. 
화면의 도식은 LangChain의 핵심 파이프라인입니다.
첫째, 사용자의 원시 입력(User Input)은 PromptTemplate을 거쳐 모델이 이해할 수 있는 문맥이 부여된 프롬프트로 변환됩니다.
둘째, 이 데이터는 LLM을 통과하며 텍스트나 메시지 객체 형태의 응답을 생성합니다.
셋째, 마지막으로 OutputParser가 이 비정형 텍스트를 어플리케이션이 요구하는 구조화된 데이터(Parsed Output)로 파싱합니다.

핵심은, 이 각기 다른 역할을 수행하는 모듈들이 어떻게 에러 없이 매끄럽게 파이프라인으로 연결되는가입니다. 다음 슬라이드부터 이 다양한 블록들을 레고처럼 결합할 수 있게 해주는 랭체인의 표준 규격, 'Runnable 인터페이스'에 대해 살펴보겠습니다.

Runnable : 랭체인의 핵심 컴포넌트들은 모두 Runnable 인터페이스를 구현합니다. Runnable interface는 Standard interface + 비동기 interface. invoke(), stream(), batch()를 포함합니다.따라서 랭체인의 핵심 컴포넌트들을 각각의 복잡한 구현체가 아닌 Runnable로 추상화 함으로써 다형성을 구현합니다.
이를 통해 레고와 같이 손쉬운 모듈 교체가 가능해집니다.
이러한 Runnable 컴포넌트들은 LCEL(LangChain Expression Language)이라는 파이프라인 문법(|)을 통해 결합됩니다.

Pipeline I/O: 
랭체인의 파이프라인은 단순한 딕셔너리의 전달처럼 보이기도 합니다. 그러나 랭체인은 각 러너블 파이프라인을 통과할 때마다 데이터를 적절한 객체로 변환시켜 전달합니다.
- 초기 입력은 dict 타입이지만, 프롬프트 템플릿을 통과하면 PromptValue로, model을 통과하면 응답 텍스트와 메타 데이터를 포함한 AIMessage 객체입니다.

OutputParser와 타입 안정성
우리가 LLM을 사용할 때에 문자열 응답만 바라고 사용하진 않습니다. 구조화된 스키마를 바탕으로 한 응답을 원하기도 합니다. 
이를 위해 사용할 수 있는 것이 OutputParser 입니다. 이 OutputParser는 LLM에게 적절한 프롬프팅을 통해 원하는 데이터 형태로 응답하라고 명령하는 컴포넌트 입니다. 중요한 점은 '프롬프팅을 통한 부탁'이라는 점입니다. 결국 outputparser는 출력의 타입 안정성을 보장하지 못합니다.

with_structured_output & Pydantic : 최근의 랭체인은 프롬프팅 의존도를 낮추고, 모델 네이티브 수준의 Function calling을 사용하도록 변경하였습니다. with_structured_output과 Pydantic을 사용하여 매우 높은 수준의 타입 안정성을 보장합니다.

PromptTemplate
프롬프트 템플릿의 핵심 기능은 사용자의 입력 등을 동적으로 렌더링하여 매 수행마다 다른 입력이 가능케하는 것입니다.
이외에도 입력받을 변수 스키마만 정의함으로써 길고 복잡한 프롬프트를 별도의 모듈 및 섹션에서 관리함을 통해 관심사의 분리를,
역할 기반 메시지 구조를 사용함으로써 LLM이 자신의 페르소나, 맥락, 사용자의 의도 등을 좀 더 용이하게 인식하도록 합니다.

1. 사용자의 입력 혹은 DB검색 결과를 주입받아 렌더링하여 동적 입력을 보장.
2. 관심사의 분리 : 프롬프트 템플릿은 입력받을 변수의 스키마만 정의함으로써 프롬프트와 비즈니스 로직의 분리를 보장함. 
3. 학습 과정에서 사용된 역할 기반 메시지 구조를 사용함으로써 LLM이 자신의 페르소나, 맥락, 사용자의 의도 등을 용이하게 인식하도록 함.[('system','당신은 신뢰도 있는 답변자입니다 ...'), ('ai','안녕하세요 {username}님'),('human','{question}')] 

정리하자면, Prompt Template은 입력의 동적 렌더링으로 확장성을 확보하고, LLM에 최적화된 메시지 구조를 강제하며, Runnable 인터페이스를 따름으로써  표준화된 파이프라인 통합을 가능하게 한다.

Models 
랭체인의 Model은 랭체인 자체 모델이 아니다. OpenAI, Google, Anthropic 등의 서드파티 AI와의 API 통신을 규격화한 어댑터이다. 
각 기업마다 API 호출 방식, 파라미터, 응답 페이로드가 다름에도 불구하고 Model 인터페이스를 통해 완전히 추상화된다. 따라서 개발자는 Model 교체 시 비즈니스 로직의 수정 없이 '객체 선언부'만을 변경함으로써 Model을 손쉽게 교체할 수 있다.


How : LCEL(LangChain Expression Language)
마지막으로 LCEL을 통해 복잡한 호출이 아닌, 선언적으로 설계된 단 한줄의 체인을 보시겠습니다.
LCEL은 파이프 연산자 오버로딩을 통해 개별 Runnable 객체들을 결합하는 선언적 프로그래밍을 따릅니다.
결과적으로 개발자는 복잡한 제어 흐름을 절차적으로 짤 필요 없습니다. 데이터가 흐를 경로만 선언하면, 비동기 처리나 병렬 실행같은 복잡한 로직은 프레임워크가 완벽히 캡슐화 합니다. 


오늘 우리는 LCEL을 통해 LLM을 제어하는 강력한 파이프라인을 구축할 수 있다는 것을 배웠습니다.
그러나 이러한 단순 프롬프트-llm 체인으론 지식 컷오프(Knowledge Cut-off)로 인한 할루시네이션을 해결할 수 없습니다.
이 파이프라인에 학교 내규나 당장 오늘의 날씨를 묻는다면 대답을 잘 할까요? 그렇지 않을 겁니다.

다음 기회가 된다면, 이를 해결할 검색 증강 생성 RAG 아키텍처에 대해 알아보겠습니다. 감사합니다.
