1. 랭체인이란?

 1) 기존 LLM 활용의 한계 및 극복 방안의 필요성 => a) Knowledge Cut off & Hallucination

자기 회귀 : 자신의 과거 출력이 미래 출력을 결정하는 방식. 예를 들어 첫 단어가 "나는"이라면 뒤에는 "~을"과 같은 목적어가 높은 확률값을 가질 것이며, 이를 기반으로 "너를" 이라는 단어를 선택, "나는 너를" 기반으로 "~하다" 라는 동사가 높은 확률값을 가질 것이며 최종적으로 "나는 너를 좋아한다" 라는 문장이 완성됨. 자기 회귀 모델은 논리적 추론이 아니라 "확률적 연결"에 의한 텍스트 생성기라고 할 수 있음.

기존의 사용 방식 : LLM이 QKV 행렬을 거쳐하여 단어와 단어 간 관계를 파악하고 자기회귀적으로 생성한다 한들 결국엔 텍스트 생성기에 불과하다는 한계를 지님.
따라서 맥락 배경 없이 LLM을 활용할 경우, 예를 들어 오늘은 구름이 꼈지만 쨍쩅한 여름인 상황에서 LLM에게 '오늘은 구름이 많네, 오늘 날씨가 어떻게 될까?' 라고 질문한다면 LLM의 사전 학습된 지식(pre-trained knowledge)에 기반하여 '구름이 많다 => 비가 올 확률이 높다'로 연결되어 실제 날씨 정보와 관련 없이 그저 그럴듯한 답을 내놓을 뿐임. 

---------------
1. 자기 회귀(Autoregressive) 모델의 본질
LLM은 과거의 출력을 다음 출력의 입력으로 사용하는 자기 회귀 방식을 취합니다. 예를 들어 "나는" 뒤에 "너를", 그 뒤에 "좋아한다"가 올 확률을 계산하여 문장을 완성합니다. 즉, LLM은 고차원적인 논리 추론 엔진이라기보다, 학습된 데이터에 기반한 **'확률적 연결 시스템'**에 가깝습니다.

2. 기존 활용 방식의 한계점: 맥락맹(Context Blindness)
LLM이 Attention 메커니즘(QKV)을 통해 문맥을 파악하고 자기 회귀로 답변을 생성하더라도, 이는 결국 모델 내부에 고정된 지식에만 의존한다는 한계를 지닙니다.사례: 맑지만 구름이 낀 여름날, "오늘 구름이 많네, 날씨가 어떨까?"라고 질문할 경우.LLM의 반응: 실시간 기상 데이터가 없는 LLM은 학습 데이터상의 통계적 상관관계("구름이 많다" $\rightarrow$ "비가 올 확률이 높다")에 매몰되어, 실제 외부 상황과 무관한 '그럴듯한 오답(Hallucination)'을 출력하게 됩니다.3. 결론: 외부 데이터와의 결합 필요성이러한 '확률적 생성'의 한계를 극복하고 실시간 맥락을 반영하기 위해서는 외부 도구 및 데이터와 LLM을 유기적으로 연결하는 프레임워크가 필수적입니다.


----------------------

이러한 특징들로 인해 기존 LLM의 활용에도 한계가 있을 수 밖에 없다.
앞서 언급한 정보의 단절, 할루시네이션 뿐만 아니라 기억력 부재 = (직접 프롬프트로 주입하지 않는 이상)이전 대화를 기억하지 못하며, 행동 능력의 부재(최신 검색 결과를 찾아줘 => 실제로 최근 검색 결과를 검색할 수 없으며, 자신의 지식=가중치 에 기반하여 그럴듯한 결과를 제공 => 할루시네이션).

이러한 한계에도 불구하고 LLM의 강력한 '자연어 이해 및 추론 능력'을 활용하기 위해 새로운 패러다임이 등장. 
ReAct(Resoning & Acting, LLM이 스스로 생각하고 행동). 이를 구현한 프레임워크 중 하나가 랭체인 생태계.

정리하자면, 랭체인이란 LLM의 한계를 극복하고, 이를 활용한 애플리케이션 개발에 특화된 프레임 워크라고 할 수 있습니다. 

그럼에도 불구하고 LLM은 높은 활용성을 보였기에 이를 활용하고자 ReAct(Reasoning & Acting) 패러다임이 제시되었고 이를 위한 프레임워크 중 하나가 랭체인 입니다.

따라서 랭체인이란, ReAct의 원리를 활용하여 LLM의 한계를 극복할 뿐만 아니라, 이를 활용한 애플리케이션 개발에 특화된 프레임워크라고 할 수 있습니다.

Why?(왜 랭체인이 등장하게 되었는가)

배경 설명(BackGround)
1) LLM 작동 방식&원리(simple ver)
  텍스트의 이해: Attention(QKV)를 활용한 트랜스포머 모델 기반. 
  예시 : ShinWon has a dog. His dog bites someone's dog.
  이러한 문장이 있을 때, 어텐션이 없다면 대명사 His가 가리키는 존재가 누구인지 알지 못한다. 마찬가지로 문장 내의 dog는 모두 같은 '개'라는 의미일 뿐, 동사의 목적어가 되는 dog와 주어가 되는 dog 도 구분하지 못한다.
  Attention은 이러한 맥락 문제를 Query Key Value 행렬과 내적을 통해 해결한다.
  즉, Attention은 LLM에게 텍스트 이해력을 부여한다.
  
  텍스트의 생성: AutoRegressive
  자신의 과거 출력이 미래 출력을 결정하는 방식. 예를 들어 첫 단어가 "나는"이라면 뒤에는 "~을"과 같은 목적어가 높은 확률값을 가질 것이며, 이를 기반으로 "너를" 이라는 단어를 선택, "나는 너를" 다시 입력하며, 이어서 "~하다" 라는 동사가 높은 확률값을 가질 것이다. 최종적으로 "나는 너를 좋아한다" 라는 문장이 완성됨. 자기 회귀 모델은 논리적 추론이 아니라 "확률적 연결"에 의한 텍스트 생성기라고 할 수 있음.
  즉, AutoRegressive는 LLM에게 텍스트 생성력을 부여한다.
  
2) LLM은 결국 "그럴듯한 텍스트 생성기"

 구름 낀 여름 하늘날을 보며, LLM에게 맥락 배경 없이, LLM에게 '오늘은 구름이 많네, 오늘 날씨가 어떻게 될까?' 라고 질문한다면 LLM의 사전 학습된 지식(pre-trained knowledge)에 기반하여 '구름이 많다 => 비가 올 확률이 높다'로 연결되어 실제 날씨 정보와 관련 없이 그저 그럴듯한 답을 내놓을 뿐임. 

  결론적으로 LLM은 고도의 논리 엔진이 아니다. 앞서 서술하였듯, 과거의 단어를 바탕으로 다음 단어가 올 "확률"을 계산하여 이어 붙이는 그럴듯한 텍스트 생성기이다.

 즉, 다음과 같은 한계를 지닌다
 - 정보의 단절(Cut-off 존재) : 모델의 가중치는 자신이 학습한 데이터에 한정된다. 최신의 정보는 반영하지 못한다.
 - 행동 능력의 부재 : LLM은 텍스트만 생성할 뿐 실제로 DB에 쿼리를 날리거나 웹에 검색할 수 있는 능력이 없다.
 - Stateless : LLM은 매번 단어 확률을 게산할 뿐, 이전 대화의 맥락을 자체적으로 기억하지 못한다.
 최종적으로 이러한 한계들은 Hallucination 현상의 발현으로 이어진다.
 
3) Why LangChain?

 LLM은 그 자체로 뛰어난 자연어 이해 및 추론 능력을 갖추고 있으나, 앞서 언급한 구조적 결함으로 인해 Standalone 시스템으론 한계를 가진다. 이를 극복하고 활용하기 위해서 RAG(Retrieval Augmented Generation), ReAct(Reasoning & Acting) 와 같은 여러 패러다임이 제시되었으며 이를 일관된 인터페이스로 구현할 수 있도록 돕는 것이 LangChain 프레임워크이다.
 
 LangChain 프레임워크는 문맥 기억, 최신 혹은 외부 정보의 주입을 용이하게 할 뿐만 아니라 Chain 기반의 연결을 통해 단계적 사고를 가능케 한다.
 LangChain 프레임워크 위에서 LLM은 더 이상 확률적 텍스트 생성기가 아니라, 외부 환경과 상호작용하며 복잡한 태스크를 수행하는 논리적인 추론 엔진으로 작동할 수 있다.
 



What? 랭체인의 주요 구성요소


Runnable : 랭체인의 핵심 컴포넌트들은 모두 Runnable 인터페이스를 따릅니다. Runnable interface는 Standard interface + 비동기 interface. invoke(), stream(), batch()를 포함합니다.따라서 랭체인의 핵심 컴포넌트는 내부 로직이 달라도 invoke() 등의 메서드를 동일하게 호출하여 다형성을 보장합니다.
파이프 연산자 오버로딩 : 뿐만 아니라 파이프 연산자 오버로딩을 통해 독립적으로 개발된 랭체인 컴포넌트들을 레고 블럭처럼 손쉽게 조립하고 해체하는 선언적 파이프라인(LCEL)구축이 가능해집니다.



초기 입력은 dict 타입이며, 러너블 파이프라인을 통과할 때마다 적절한 랭체인 제공 객체로 패키징됩니다. 
최종 출력은 응답 텍스트와 여러 메타데이터를 포함한 AIMessage 객체이지만, 다양한 OutputParser를 사용하여 출력을 규격화할 수 있습니다.
이 OutputParser는 프롬프팅을 활용한 방법이므로, 타입에 대한 확신이 없었습니다만 최근엔 with_structured_output 메서드를 통해 지정된 Pydantic 객체로 출력하도록 강제함으로써 타입 안정성을 확보하였습니다.



Model I/O: 
랭체인의 파이프라인은 단순한 딕셔너리의 전달처럼 보이기도 합니다. 그러나 랭체인은 각 러너블 파이프라인을 통과할 때마다 데이터를 적절한 객체로 변환시켜 전달합니다.
- 초기 입력은 dict 타입이지만, 프롬프트 템플릿을 통과하면 PromptValue로, model을 통과하면 응답 텍스트와 메타 데이터를 포함한 AIMessage 객체입니다.

OutputParser와 타입 안정성
가장 중요한 것은 비정형 데이터인 응답 텍스트를 OutputParser&with_structured_output을 통해 정형 데이터로 직렬화 할 수 있다는 점입니다.
초기 : 초기의 OutputParser는 LLM에게 "JSON 형식으로 대답해"라고 프롬프팅한 뒤 텍스트를 파싱하는 방법이었으므로, 출력 타입의 안정성이 보장되지 않았습니다.
with_structured_output & Pydantic : 최근의 랭체인은 프롬프팅 의존도를 낮추고, 모델 네이티브 수준의 Function calling을 사용하도록 변경하였습니다. with_structured_output과 Pydantic을 사용하여 완벽한 타입 안정성을 보장합니다.

PromptTemplate
1. 관심사의 분리 : 프롬프트 템플릿은 입력받을 변수의 스키마만 정의함으로써 프롬프트와 비즈니스 로직의 분리를 보장함.
2. 사용자의 입력 혹은 DB검색 결과를 주입받아 렌더링하여 동적 입력을 보장.
3. 학습 과정에서 사용된 역할 기반 메시지 구조를 사용함으로써 LLM이 자신의 페르소나, 맥락, 사용자의 의도 등을 용이하게 인식하도록 함.[('system','당신은 신뢰도 있는 답변자입니다 ...'), ('ai','안녕하세요 {username}님'),('human','{question}')] 

정리하자면, Prompt Template은 입력의 동적 렌더링으로 확장성을 확보하고, LLM에 최적화된 메시지 구조를 강제하며, Runnable 인터페이스를 따름으로써  표준화된 파이프라인 통합을 가능하게 한다.

Models 
랭체인의 Model은 랭체인 자체 모델이 아니다. OpenAI, Google, Anthropic 등의 서드파티 AI와의 API 통신을 규격화한 어댑터이다. 
각 기업마다 API 호출 방식, 파라미터, 응답 JSON 구조가 다름에도 불구하고 Model 인터페이스를 통해 완전히 추상화된다. 따라서 사용자는 Model 교체 시 비즈니스 로직의 수정 없이 '객체 선언부'만을 변경함으로써 Model을 손쉽게 교체할 수 있다.
